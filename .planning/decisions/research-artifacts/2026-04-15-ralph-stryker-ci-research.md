# How top engineers harden CI/CD for AI agent code

**The best engineering teams don't trust AI-generated code — they build systems that make trust unnecessary.** Stripe ships 1,300+ AI-generated pull requests per week. OpenAI built a million-line codebase with zero human-written code. Shopify routes every AI PR through canary deployments serving 5% of live traffic. These companies share a common insight: the quality of AI-generated code depends less on the AI model and more on the *pipeline infrastructure* surrounding it. A new discipline called "harness engineering" has emerged to describe this shift — engineers no longer write code; they design the constraints, feedback loops, and quality gates that make AI agents reliably productive. This report breaks down exactly how they do it, with real pipeline designs, actual workflow examples, and practical tools you can adopt today.

## Stripe's Minions: the gold standard in agent pipeline design

Stripe's internal system, **Minions**, generates over **1,300 pull requests per week** containing zero human-written code. Built by the "Leverage" team and publicly detailed in February 2026, it represents the most mature production-grade AI agent pipeline documented to date. The system is built on a **"Blueprints" architecture** that alternates between hardcoded deterministic steps and free-flowing agentic LLM steps — the critical insight being that the AI agent can never skip required quality checks.

The pipeline flow works like this: a Slack emoji reaction triggers a Minion, which spins up a pre-warmed **sandboxed AWS EC2 devbox** in under 10 seconds. Each devbox is identical, disposable, and isolated from production — the agent gets full shell permissions because the blast radius is contained. The agent then plans and writes code using a heavily modified fork of Block's open-source **Goose** coding agent, optimized for fully unattended operation. Once coding is complete, the pipeline enforces a **three-tier feedback loop** that represents the core of Stripe's quality gate design.

**Tier 1** runs local linters in under 5 seconds — a background daemon precomputes which lint rules apply, including **Sorbet type-checking** across 15 million+ lines of Ruby. **Tier 2** triggers selective CI that runs only the tests relevant to changed files — roughly **5% of Stripe's 3+ million tests**, orchestrated by Bazel. **Tier 3** enforces a hard cap: if the agent can't fix CI failures in **two attempts**, it escalates to a human engineer. This cap prevents infinite retry loops that burn compute and tokens without converging on a solution. Even partially complete PRs (~80% done) are submitted as valuable starting points.

Two design principles define Stripe's approach. First, the agent has **submission authority, not merge authority** — every single PR is reviewed by a human before merging. Second, context is scoped surgically: **directory-level rule files** attach automatically as the agent navigates the filesystem, preventing context window overflow. Rather than dumping a massive instruction file, Stripe adopted Cursor's rule file format and synchronized it across Minions, Cursor, and Claude Code, so guidance written for one system works everywhere. The agent accesses approximately **500 internal tools** through a centralized MCP (Model Context Protocol) server called "Toolshed," but each task loads only a curated subset of roughly 15 relevant tools.

## Shopify routes AI code through canary deployments and human review

Shopify operates the world's largest Ruby on Rails monorepo, and their approach to AI code quality is built on a foundation of existing infrastructure rather than AI-specific tooling. **Every AI-generated PR requires senior engineer review before merge — no exceptions, no autonomous commits.** As Farhan Thawar, VP of Engineering, stated plainly: "Shopify is not yet at the place where we allow AI to check in code automatically into the repos."

The release pipeline that all code — human and AI alike — must pass through runs end-to-end in approximately 15 minutes. After a developer triggers `/shipit`, the PR enters a custom-built **merge queue** that integrates it with the trunk branch and runs **55,000+ tests**. Changes then deploy to **canary infrastructure receiving 5% of all incoming requests** with a mandatory 10-minute observation window and automated canary analysis. If no alerts fire, changes promote to production. Shopify tracks **reversion rates** as the primary quality signal and has observed no increase since AI tool adoption, despite a measurable increase in PR volume.

Shopify's most distinctive contribution is **Roast**, an open-source workflow orchestration framework built by their Augmented Engineering DX team. Roast addresses a specific problem: "Allowing AI to roam free around millions of lines of code just didn't work very well. Non-determinism is the enemy of reliability." Roast uses YAML configuration with markdown prompts to interleave **deterministic code execution** with non-deterministic AI behavior, making AI workflows version-controlled, testable, and CI/CD-integrable. One production workflow, **Boba**, automatically adds Sorbet type annotations to test files through a loop: clean up with `sed`, bump to strict typing, run Sorbet autocorrect, feed remaining issues to a Claude Code agent, then iterate until tests and type checking pass. Another workflow grades test quality and generates replacements for low-scoring tests — creating a self-improving test suite.

Daniel Doubrovkine, Director of Augmented Engineering, articulated the strategic logic: "We want to use AI to generate high-quality missing tests. Why? Ultimately, we want AI agents to refactor our code safely, and for that, we need robust test coverage." This reveals Shopify's layered approach — use AI to strengthen the safety net, then use that safety net to enable more aggressive AI contributions.

## Harness engineering: the emerging discipline of constraining AI agents

The term **"harness engineering"** has crystallized as the industry's name for designing environments that make AI agents reliable. Mitchell Hashimoto, creator of Terraform, coined the concept in his widely-shared AI adoption journey: **"Every time you discover an agent has made a mistake, you take the time to engineer a solution so that it can never make that mistake again."** This compounds — every improvement applies to every future agent run.

OpenAI validated the concept at scale. A three-person team started with an empty repository in August 2025 and built an internal beta product over five months — **one million lines of code, 1,500 merged PRs, zero human-written code**. They estimated it took one-tenth the time of manual development. The team built "Symphony," an Elixir-based orchestration system managing multiple Codex agents, and learned four key lessons that define harness engineering in practice.

First, **context management trumps model capability**. OpenAI tried the "one big AGENTS.md" approach and it failed — a giant instruction file crowds out the task, the code, and relevant documentation. Instead, they built a map: scoped documentation, directory-level conventions, and dynamic context providers. Second, **architectural constraints must be enforced mechanically**, not just documented — using deterministic custom linters and structural tests (like ArchUnit) rather than relying on the AI to remember rules. Third, **"garbage collection" agents** run periodically to find inconsistencies in documentation or architectural violations, fighting the entropy that accumulates as agents generate code. Fourth, observability data — logs, metrics, and spans — feeds directly into agent context, letting agents monitor and debug their own output.

Birgitta Böckeler, Distinguished Engineer at Thoughtworks, published a rigorous analysis on Martin Fowler's site that formalized harness components into two dimensions. **Guides** steer agent behavior (computational: linters, structural tests; inferential: AGENTS.md, knowledge bases). **Sensors** detect quality issues (computational: unit tests, fitness functions; inferential: LLM-based code review). The key insight is that harnesses "attempt to externalise and make explicit what human developer experience brings to the table." They are not about removing humans but about directing human attention to where it matters most.

## How Vercel, Netflix, and others adapted their pipelines

Vercel's engineering team published one of the most candid assessments of AI code risk. Their core thesis: **"Green CI is no longer proof of safety."** Passing CI merely reflects the agent's ability to persuade your pipeline that a change is safe — even when a flawless-looking PR ships a query that scans every row in production, or retry logic that causes a thundering herd on a downstream service. Over **30% of Vercel's deployments** are now initiated by coding agents, up 1,000% in six months.

Vercel's response includes several concrete pipeline innovations. **Self-driving deployments** roll out every change incrementally through gated pipelines — if a canary degrades, the rollout stops and rolls back automatically without requiring an engineer to babysit a dashboard. **Executable guardrails** encode operational knowledge as runnable tools rather than documentation: a `safe-rollout` skill wires a feature flag, generates a rollout plan with rollback conditions, and specifies how to verify expected behavior. Most distinctively, Vercel deploys **read-only audit agents** that continuously verify system invariants in production — specialized agents whose sole job is to check the assumptions made by generative agents. They also track **defect-commit versus defect-escape ratios** to surface when risk is quietly increasing.

Netflix adapted their ML/AI development infrastructure for agent workflows by teaching agents to use faster development loops. Their `spin` tool surfaces errors faster and contextualizes them to specific code pieces, increasing the chance agents can fix errors autonomously. Their CLAUDE.md files include incremental development workflow instructions. The broader Netflix deployment stack — **Metaflow** for ML pipelines, **Maestro** for orchestration, and **Spinnaker** for canary deployments — provides the safety net that makes AI contributions manageable across 3,000+ developers.

Cloudflare took a sandboxing approach with **Dynamic Workers** — isolate-based runtimes executing AI-generated code in sandboxed environments that start 100x faster than containers and scale to millions of concurrent executions. Their Agents SDK includes a "Think" framework for managing long-running autonomous tasks and a "Code Mode" that generates code rather than making tool calls, providing clearer audit trails.

## Real GitHub repos and workflow files you can study

Several open-source repositories demonstrate production-grade CI/CD for AI agent workflows with actual YAML configurations.

**Browserbase's Stagehand** stands out with a 903-line CI workflow containing **12 progressive jobs** in a dependency chain: path-based change detection → ESLint and Prettier linting → TypeScript compilation with Vitest unit tests → Playwright E2E tests (both local and cloud) → AI model regression testing → category-specific evaluation suites. The evaluations are label-controlled, meaning reviewers can trigger deeper AI evaluation passes by adding GitHub labels to the PR.

**GitHub's own gh-aw** (Agentic Workflows) represents the platform's official answer — a framework that compiles natural-language markdown workflow descriptions into security-hardened `.lock.yml` GitHub Actions files. It enforces read-only permissions by default, SHA-pinned actions, and a companion **Agent Workflow Firewall** (gh-aw-firewall) for network egress control, with 13 active CI workflows including CodeQL, Trivy, and npm audit.

**AWS's sample-ai-coding-standards-template** provides an enterprise-grade starting point with pre-commit security hooks, Semgrep scanning rules, Ruff linting, hexagonal architecture enforcement, and a multi-environment CI/CD pipeline with explicit deployment gates between Development, Staging, and Production.

For AI-specific quality gates, **Kalibra** deserves attention — a purpose-built CLI tool and GitHub Action that catches hidden regressions in AI agent behavior. It compares two populations of agent traces using statistical methods (bootstrap 95% confidence intervals, z-tests), tracking 10 metrics including success rate, cost, duration, and token efficiency. Its configuration file lets you set precise thresholds: maximum 2 percentage-point success rate drop, maximum 5 regressed tasks, maximum 20% cost increase. For AI code review automation, **Qodo's PR-Agent** (open-source, formerly CodiumAI) and **CodeRabbit** both offer GitHub Actions that add AI-powered review to any PR, with CodeRabbit also integrating static analysis tools like golangci-lint, shellcheck, and yamllint.

## The composite pipeline pattern that prevents cascading failures

The specific failure mode of "AI fixes one thing, breaks another" is addressed through pipeline architecture rather than any single tool. Research from CodeRabbit and TFiR found that AI-assisted code generation produces **1.7x more issues** related to logical and correctness bugs. The industry has converged on a layered defense pattern.

**SonarQube's AI Code Assurance** introduced a dedicated quality gate specifically for projects containing AI-generated code, applying higher standards for complexity, bugs, injection vulnerabilities, and duplication. Standard thresholds enforce **≥80% coverage on new code**, zero code smells, zero security hotspots, and less than 3% duplicated lines. The system automatically detects GitHub Copilot usage and applies the stricter gate.

The most mature composite pipeline pattern combines multiple tools in a specific sequence. At PR creation, fast checks run first: linting (ESLint, Ruff), type checking (TypeScript's `tsc --noEmit`, mypy), and unit tests with coverage enforcement. Then heavier gates engage: SonarQube quality analysis, security scanning (Snyk, Semgrep, or Dependabot), and AI-powered code review (CodeRabbit or PR-Agent posting inline comments). Kalibra or similar tools evaluate agent-specific regression metrics. GitHub branch protection rules require all status checks to pass, conversation threads to be resolved, and stale approvals to be dismissed on new pushes. Only then does a human reviewer see the PR.

After merge, the code enters progressive deployment. Shopify's 5% canary with a 10-minute window, Stripe's blue-green deployments, and Vercel's self-driving rollouts with automatic rollback all serve the same purpose: **catching problems that passed every gate but only manifest under real traffic.** LaunchDarkly's AI Config pipeline adds LLM-as-judge evaluation of AI-generated configurations before they reach production, with drift detection to keep code defaults synchronized.

Thoughtworks' analysis of 28 million CI workflows revealed a sobering reality: average throughput is up 59% year-over-year, but **success rates have fallen to 70.8% — a five-year low** — and recovery times have risen to 72 minutes. Feature branch throughput surged 50% but almost none translated to shipped changes. Martin Fowler's diagnosis: "Most organizations applying AI to software delivery are still treating it as a coding tool rather than a systems transformation."

## Practical principles any builder can apply

The research converges on a set of principles that distinguish teams succeeding with AI agent code from those generating noise.

**Invest in the harness before the agent.** Stripe's core insight is that "what's good for human engineers is good for AI agents" — pre-existing investments in developer experience (documentation, devboxes, CI/CD, tooling) directly translate to higher agent success rates. Start by ensuring your project has strong test coverage, clear documentation, enforced linting, and type checking. These create the feedback loops agents need to self-correct.

**Interleave deterministic gates with agentic freedom.** Both Stripe's Blueprints and Shopify's Roast enforce the same pattern: creative work (planning, coding, problem-solving) gets full agent autonomy, but critical steps (running linters, pushing branches, checking tests) are hardcoded and unskippable. The agent cannot decide to skip the quality gate.

**Set hard limits on retry loops.** Stripe caps CI attempts at two. OpenAI's team found diminishing returns beyond a few iterations. Without explicit caps, agents will burn compute indefinitely trying to fix cascading failures. A partially complete PR that a human can finish in 20 minutes is more valuable than an agent spiraling for hours.

**Scope context surgically.** Both Stripe and OpenAI abandoned the "one giant instruction file" approach. Directory-scoped rules, task-relevant tool subsets, and progressive context loading prevent the agent from drowning in irrelevant information. Stripe curates ~15 tools per task from a library of 500.

**Treat CI as necessary but insufficient.** Vercel's warning that green CI doesn't equal safety reflects hard-won experience. Complement CI with canary deployments, production invariant checking, feature flags for gradual rollout, and automated rollback. The code that will cause an outage often passes every test — it's the interaction with production data, traffic patterns, and downstream services that creates failure.

## Conclusion

The era of "harness engineering" marks a fundamental shift in what it means to be a software engineer. The companies shipping the most AI-generated code — Stripe at 1,300+ PRs per week, OpenAI at a million lines in five months — succeed not because they have better models but because they built better constraints. The pattern is clear: sandboxed execution environments, tiered feedback loops with hard retry limits, deterministic quality gates the agent cannot skip, scoped context that prevents information overload, mandatory human review before merge, and progressive deployment with automatic rollback after merge. Kief Morris at Thoughtworks captured the scaling insight best: the only sustainable human posture is **"on the loop"** — building and maintaining the harness — rather than "in the loop" reviewing every line. The tools are available today: Roast for structured agent workflows, Kalibra for agent regression detection, SonarQube AI Code Assurance for dedicated quality gates, CodeRabbit or PR-Agent for automated review, and GitHub's Agentic Workflows framework for security-hardened pipeline generation. The bottleneck is no longer writing code. It is designing the system that keeps AI code safe.
