# Spec Runner — Execute Spec-Driven Tests

Convert spec markdown checkpoints into Playwright tests, run them against the live app, and write PASS/FAIL results back to the spec files.

Optional: $ARGUMENTS (module name to run one module, or "all" for everything in the queue)

## Why This Exists

Spec files describe WHAT to test as checkpoints. This command turns those checkpoints into executable Playwright tests, runs them, and records the results. Failed checkpoints become fix targets for the self-annealing loop.

Adapted from ScholarSync's qa/controller.ts + spec-to-playwright.ts — battle-tested across 376 specs and 12,804 checkpoints.

## Prerequisites

- Playwright installed: `npx playwright install chromium`
- App running locally (dev server)
- Spec files exist in `qa/specs/<module>/` (generated by `/census-to-specs`)

## Process

### Step 1: Check Prerequisites

```bash
# Is the app running?
APP_PORT=${APP_PORT:-3000}
curl -s -o /dev/null -w "%{http_code}" http://localhost:$APP_PORT

# If not running, start it
npm run dev &
npx wait-on http://localhost:$APP_PORT --timeout 30000
```

```bash
# Is Playwright installed?
npx playwright --version || npx playwright install chromium
```

### Step 2: Load the Queue

Read `qa/queue.jsonl` and filter by module if specified:

```bash
# If module specified
cat qa/queue.jsonl | grep "\"module\":\"$ARGUMENTS\"" | grep "\"status\":\"pending\""

# If "all"
cat qa/queue.jsonl | grep "\"status\":\"pending\""
```

### Step 3: For Each Spec — Generate and Run

For each pending spec in the queue:

#### 3a: Read the Spec File

Parse the spec markdown. Extract:
- `MODULE` — the module name
- `PAGE` — the URL to navigate to
- Every `- [ ]` line — a checkpoint to test
- Section headers (`###`, `####`) — group checkpoints logically

#### 3b: Generate the Playwright Test

Create a Playwright test file at `qa/generated/<module>/<spec-id>.spec.ts`:

```typescript
import { test, expect } from '@playwright/test';

test.describe('<module> — <spec-id>', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('<PAGE_URL>');
    await page.waitForLoadState('networkidle');
  });

  // For each checkpoint:
  test('<checkpoint description>', async ({ page }) => {
    // The test logic depends on the checkpoint type.
    // For UI element existence:
    await expect(page.getByRole('button', { name: '<button text>' })).toBeVisible();
    
    // For text content:
    await expect(page.getByText('<expected text>')).toBeVisible();
    
    // For interaction:
    await page.click('<selector>');
    await expect(page.locator('<result selector>')).toBeVisible();
    
    // For keyboard shortcuts:
    await page.keyboard.press('Control+B');
    // verify effect
  });
});
```

**Test generation strategy:**

The agent reads each checkpoint description and generates the appropriate Playwright assertion. This is where the LLM adds value — translating human-readable test descriptions into executable code. But it MUST:

- Use the accessibility tree for selectors (role, name, label) — NOT CSS classes
- Use `getByRole`, `getByText`, `getByLabel`, `getByPlaceholder` — NOT `page.locator('.some-class')`
- Handle async properly (await everything)
- Add reasonable timeouts for dynamic content
- Skip checkpoints tagged `[CODE-ONLY]` (mark as BLOCKED — not testable at runtime)

#### 3c: Run the Test

```bash
npx playwright test qa/generated/<module>/<spec-id>.spec.ts --reporter=json
```

Capture the JSON output. Parse results per test case.

#### 3d: Write Results Back to Spec

For each checkpoint in the spec file:
- Test passed → `- [x] PASS: **<feature>** — <description>`
- Test failed → `- [ ] FAIL: **<feature>** — <description>`
- Test couldn't run (element not found, page error) → `- [ ] BLOCKED: **<feature>** — <reason>`

Update the header:
```markdown
STATUS: PARTIAL (or COMPLETE if all passed)
TESTED: <N>/<total>
PASS: <N>
FAIL: <N>
BLOCKED: <N>
```

#### 3e: Update Queue

Update the queue entry in `qa/queue.jsonl`:
```json
{"id":"<module>.spec-001","status":"tested","pass":<N>,"fail":<N>,"blocked":<N>,"attempts":1}
```

```bash

```

## Test Generation Patterns

When generating Playwright tests from checkpoint descriptions, use these patterns:

**"<Element> exists / is visible":**
```typescript
await expect(page.getByRole('button', { name: /bold/i })).toBeVisible();
```

**"Click <X>, verify <Y> appears":**
```typescript
await page.getByRole('button', { name: /insert table/i }).click();
await expect(page.locator('table')).toBeVisible();
```

**"Type <text>, verify <effect>":**
```typescript
await page.getByRole('textbox').fill('Hello world');
await expect(page.getByRole('textbox')).toHaveValue('Hello world');
```

**"Keyboard shortcut <keys> does <action>":**
```typescript
await page.getByRole('textbox').focus();
await page.keyboard.press('Control+b');
// Verify the formatting was applied
```

**"Drag file onto <zone>":**
```typescript
// Use Playwright's file chooser or input[type=file]
const fileInput = page.locator('input[type="file"]');
await fileInput.setInputFiles('test-file.pdf');
```

**"Error state when <condition>":**
```typescript
// Trigger the error condition
await page.route('**/api/endpoint', route => route.abort());
await page.getByRole('button', { name: /submit/i }).click();
await expect(page.getByText(/error/i)).toBeVisible();
```

## Rules

- NEVER skip failed tests — they become fix targets for annealing
- NEVER mark a test as PASS without Playwright confirmation — evidence before claims
- Use accessibility-first selectors (getByRole, getByText) — CSS classes are brittle
- If a checkpoint can't be automated (requires human judgment), mark BLOCKED with reason
- Keep generated test files — they're reusable and version-controlled
- Run with `--reporter=json` for machine-parseable results
